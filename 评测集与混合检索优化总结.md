# 评测集与混合检索优化总结

## 1. 背景与目标

ResearchCode-Agent 原有的单元测试和集成测试主要验证接口、扫描器、检索服务和 Agent 工作流能否正常运行，但不能回答以下质量问题：

- 混合检索是否真的优于纯向量检索；
- 关键词检索是否提供了互补候选，还是已经退化成无效分支；
- `0.7/0.3` 的向量/关键词权重是否只是经验值；
- Agent 返回的文件和代码证据是否来自真实源码；
- 中文、英文和不同任务类型上的效果是否一致。

因此，这一阶段的目标不是单纯增加几个测试用例，而是建立一套可复现、可审计的代码理解评测体系，并用它驱动混合检索优化。

## 2. 真实仓库与冻结快照

评测选用了真实开源项目 RuoYi-Vue，而不是只依赖项目内的小型示例代码。

- 仓库 Commit：`41720e624c5a668c7d3777835e4c87095a7a1dfd`
- 支持的源文件：约 450 个
- 代码行数：约 5.8 万行
- 扫描得到的代码实体：2,199 个
- 静态关系：1,980 条
- 解析错误：0

实验固定仓库 Commit、SQLite 数据、Qdrant 快照、评测集哈希和配置指纹。这样可以避免仓库更新、索引重建或配置变化导致前后实验不可比较。

外部仓库源码、SQLite、Qdrant 数据和 Provider 运行账本位于忽略目录中，不提交到 Git；可复现所需的 Schema、数据集、脚本、指标和报告保存在 `evaluation/` 下。

## 3. Pilot 测试集

当前正式 Pilot 包含 36 道问题：

| 任务类型 | 数量 | 评测内容 |
| --- | ---: | --- |
| CODE_QA | 12 | 功能、类、方法或接口在哪里实现 |
| TRACE_CHAIN | 12 | 前端请求如何追踪到后端接口和服务 |
| CHANGE_PLAN | 12 | 修改某项功能可能影响哪些文件 |

分布情况：

- 中文 18 道，英文 18 道；
- Easy 8 道、Medium 16 道、Hard 12 道；
- `source_answerable=true` 27 道；
- `source_answerable=false` 9 道。

另有 6 道基于历史 Commit 的变更题。这些题具有独立的 base/target SHA，被放在 `historical-prepared` 中，没有混入当前 HEAD 指标，避免使用未来版本代码回答历史问题。

## 4. 结构化 Gold 与答案边界

每道题不仅保存自然语言问题，还保存结构化 Gold：

- 必须召回的文件和代码实体；
- 相关图节点和静态关系；
- 文件路径与代码行号证据；
- 实体是否真实存在于冻结源码；
- 当前扫描器是否能够索引该事实；
- 语言、难度、任务类型和审核状态。

数据模型特意区分：

- `source_answerable`：源码本身是否包含充分答案；
- `system_answerable`：当前扫描、索引和静态图是否能观察到这些答案。

例如 Controller 对 Service 的调用在源码中真实存在，但旧扫描器可能因为接口类型和实现类名称不一致而没有建立 `CALLS_METHOD` 边。此时问题在源码层面可回答，但系统证据不足。这个区分可以判断失败究竟来自题目、检索还是静态分析能力。

## 5. 数据质量与审核流程

为避免同一个 Agent 出题、标答案并给自己打分，评测流程拆分为不同职责：

1. Question Curator 负责选题；
2. Evidence Annotator 负责从源码提取证据；
3. Adversarial Reviewer 对标注进行对抗审查；
4. Adjudicator 处理分歧；
5. 确定性验证器检查文件、实体、关系、语言和快照身份。

旧的非法格式数据和受污染指标没有被直接覆盖，而是归档并标记为 superseded。当前 Pilot 达到 36/36 Schema 和语义验证通过。

## 6. 指标定义

检索指标使用文件级 fractional Recall，而不是更宽松的“命中任意一个文件就算成功”。

如果一道题要求召回 4 个文件，而 Top 5 找到其中 3 个，则该题的 Recall@5 为 `3/4=0.75`。

主要指标包括：

- Recall@1：第一个结果是否覆盖正确证据；
- Recall@5：有限上下文预算下的核心证据覆盖；
- Recall@10：候选集合的覆盖上限；
- MRR：第一个正确结果的平均倒数排名；
- Zero-hit rate：完全没有命中 Gold 的比例。

Recall@10 单独使用会比较宽松，因此最终同时报告 Recall@1、Recall@5、Recall@10 和 MRR。

## 7. 发现的混合检索退化问题

最初的关键词检索直接处理自然语言查询，存在以下问题：

- 中文问题与英文代码标识符难以直接匹配；
- query rewrite 中仍包含较多自然语言噪声；
- camelCase、API 路径、类名和方法名没有被充分利用；
- 关键词分支可能返回大量泛化结果；
- 混合检索虽然执行了两个分支，但关键词分支对最终结果贡献有限。

如果在这个状态下直接调整权重，只是在重新分配一个质量不足的信号。因此优化顺序是先修复关键词分支，再校准融合比例。

## 8. 代码感知的关键词检索修复

新增了代码感知的 lexical query 处理，从自然语言中提取有价值的代码检索词，包括：

- 类名和方法名；
- camelCase 标识符片段；
- API 路径片段；
- Java/Vue 常见代码符号；
- 有界的功能关键词。

同时限制无意义的自然语言词和过宽查询，减少关键词分支把泛化词面匹配排在真正实现证据之前的情况。

修复后，在同一套 27 道源码可回答检索题上：

| 方法 | Recall@1 | Recall@5 | Recall@10 | MRR |
| --- | ---: | ---: | ---: | ---: |
| B2 纯向量检索 | 52.78% | 93.52% | 98.15% | 82.10% |
| B3 混合检索 0.70/0.30 | 56.48% | 97.22% | 98.15% | 84.88% |

混合检索相较纯向量检索：

- Recall@5 提升 3.70 个百分点；
- Recall@1 提升 3.70 个百分点；
- MRR 提升 2.78 个百分点；
- Recall@10 保持不变。

这证明修复后的关键词分支可以提供不同候选，并在前五名范围内补充纯向量检索遗漏的证据。

## 9. 融合比例实验

原来的 `0.7/0.3` 是工程经验值。为了给这个选择提供依据，实验固定数据集、快照、查询改写、候选生成配置、Embedding 模型、Top-K 和评分脚本，只改变向量与关键词权重。

实验包含纯向量 B2，以及以下向量权重：

`0.60、0.65、0.70、0.75、0.80、0.85、0.90、0.95`

对应关键词权重为 `1 - vector_weight`。总运行量为：

`36 道问题 × 9 个配置 = 324 条真实检索记录`

候选比例必须满足预先定义的门禁：

1. Recall@10 不低于 98.15%；
2. Recall@5 不低于旧配置的 97.22%；
3. Recall@5 或 MRR 至少一项严格优于 0.70/0.30；
4. 中文、英文和三个任务类型的 Recall@5 均不得退化。

## 10. 权重实验结果

| 向量/关键词比例 | Recall@1 | Recall@5 | Recall@10 | MRR |
| --- | ---: | ---: | ---: | ---: |
| 0.60/0.40 | 56.48% | 97.22% | 98.15% | 84.88% |
| 0.65/0.35 | 56.48% | 97.22% | 98.15% | 84.88% |
| 0.70/0.30 | 56.48% | 97.22% | 98.15% | 84.88% |
| **0.75/0.25** | **60.19%** | **97.22%** | **98.15%** | **86.73%** |
| 0.80/0.20 | 60.19% | 97.22% | 98.15% | 86.73% |
| 0.85/0.15 | 60.19% | 97.22% | 98.15% | 86.73% |
| 0.90/0.10 | 60.19% | 97.22% | 98.15% | 86.73% |
| 0.95/0.05 | 45.37% | 97.22% | 98.15% | 80.25% |

最终生产默认值从 `0.70/0.30` 调整为 `0.75/0.25`。

`0.75–0.90` 构成同分平台，选择 0.75 是因为它距离旧配置最近，属于平台内最小改动，而不是因为实验能证明 0.75 是一个精确的普适最优点。

`0.95/0.05` 的 Recall@1 和 MRR 明显下降，也说明关键词分支仍然有价值，不能简单退化为接近纯向量检索。

## 11. 排序提升来自哪里

0.75/0.25 相较 0.70/0.30 的提升集中在一道 CODE_QA：

> Where is the department tree structure API implemented for the user management page?

在旧比例下，前端 `user.js` 排在第一，正确的后端 `SysUserController` 排在第二。新比例略微降低词面匹配影响后，后端实现上升到第一。

因此：

- Recall@1：`56.48% → 60.19%`；
- MRR：`84.88% → 86.73%`；
- Recall@5、Recall@10 均不退化。

这个案例说明关键词分支适合补充精确符号和路径，但权重过高时可能把前端调用包装器排在真正的后端实现之前。

## 12. 独立审计与验证

权重结论由独立审查 Agent 从原始 324 条记录和 Gold 中重新计算，核验了：

- 数据集、Commit、DB、Snapshot 和配置指纹；
- 每个比例的 Recall@1/5/10 与 MRR；
- 中文、英文和任务类型分组是否退化；
- Provider ledger 和缓存来源；
- 是否存在 Gold 泄漏、题目 ID 特判或结果硬编码；
- Bootstrap 和 leave-one-out 结果。

生产代码通过 TDD 修改，并经过独立规格审查和代码质量审查。最终验证包括：

- evaluation 测试：402 passed；
- backend 全量测试：退出码 0，共收集 298 条，1 条 skipped；
- 融合单元/API 测试：17 passed；
- `compileall`：通过；
- `git diff --check`：通过。

## 13. 可以得出的结论与边界

可以得出的结论：

> 在冻结的 RuoYi-Vue development Pilot 上，修复后的混合检索将 Recall@5 从 93.52% 提升到 97.22%，同时保持 Recall@10 为 98.15%。在固定变量的权重网格实验中，0.75/0.25 相较 0.70/0.30 保持 Recall@5/10 不退化，并将 Recall@1 从 56.48% 提升到 60.19%、MRR 从 84.88% 提升到 86.73%。

不能据此声称：

- 0.75/0.25 是所有仓库的通用最优值；
- 当前提升具有统计显著性；
- 已证明跨语言、跨框架泛化；
- 每一道问题都因新比例得到改善。

比例提升来自 27 道可评问题中的一道。Leave-one-out 分析显示，删除这道题后新旧比例没有胜者。因此当前结果应描述为“有数据依据的单仓开发集工程校准”，下一步需要在更多仓库或 held-out 数据上验证。

## 14. 面试时的简洁表述

> 项目最初只有功能测试，无法证明 RAG 检索质量。我基于真实 Java/Vue 开源仓库构建了 36 道源码证据驱动的评测集，用结构化 Gold 记录必须召回的文件、实体和调用关系，并通过独立标注、对抗审查和快照校验避免 Agent 自问自判。实验发现自然语言关键词分支存在退化，于是增加代码感知的 query 处理，使混合检索 Recall@5 从 93.52% 提升到 97.22%。之后固定数据、索引、query rewrite 和候选生成配置，完成 324 次权重网格实验，将向量/关键词比例从 0.70/0.30 校准为 0.75/0.25，在 Recall@5 和 Recall@10 不退化的情况下提升 Recall@1 和 MRR。这个结果是单仓 Pilot 校准，我不会把它描述成跨项目通用最优值。
